This report is the living engineering record for SEZE. It began as a brand + UI overhaul and grew into a full-stack functionality pass: fixing a systemic persistence failure (the app wrote to a non-existent 'profiles' table), closing security and clinical-safety holes, wiring real user data across every screen, adding a nutrition data layer, an admin panel, a reviewable feedback system, and a per-programme workout engine. Every backend flow below has been verified against the live database.

06 — Key decisionsWhy it's built this way

AI/LLM deferred for the testing phase
Keep the fast local food parser + real DB lookup now; wire an LLM later. No Railway needed — the analyze-image Edge Function already exists.
Nutrition data = Open Food Facts + USDA + curated fast-food table
Free, no scraping fragility; curated table gives accurate UK/US chain items; OFF/USDA cover the long tail.
Matcher prioritises life stage over goal, but users can switch
A postnatal body shouldn't be dropped into heavy glute loading; but 'Start this programme' now lets the user override and persists it.
Profile table is rosa_users, not profiles
The app wrote to a non-existent 'profiles' table — the real table (with the signup trigger + RLS) is rosa_users. All reads/writes realigned.
Admin gated by a rosa_admins allowlist + SECURITY DEFINER RPCs
Data stays safe even if the UI leaks; add/remove admins with one SQL insert; check is case-insensitive.
Anon Supabase key in the frontend is safe
It is the public anon key; every table's RLS is auth.uid()-scoped and was verified correct.
